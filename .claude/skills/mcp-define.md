# MCP Define — 功能规范创建

创建严谨的功能说明规范文件 `specs/[feature_name]/spec.md`。

## 交付标准（6 项硬性指标）

| # | 自检项 | 合格标准 |
|---|--------|----------|
| 1 | `file:///` 可点击源码链接 | ≥ 3 处，格式 `[文件名:L行号](file:///绝对路径#L行号)` |
| 2 | ASCII 布局图 | 视觉需求章节含 ≥ 1 张布局对比图 |
| 3 | 代码片段 | ≥ 2 处 fenced code block（根因代码 + 修复代码） |
| 4 | 边界情况 | ≥ 4 条，每条含场景 + 风险 + 缓解策略 |
| 5 | 现有机制复用说明 | 明确列出复用了哪些已有变量/函数 |
| 6 | 文件清单表格 | Markdown 表格列出所有改动文件 |

## 执行指令

### 1. 加载上下文
- 阅读 `.agent/rules/project-rules.md` 确认技术栈与编码规范
- 理解用户需求，查看附带的所有截图/设计稿

### 2. 深度代码调研（≥ 5 次搜索/读取）
- 用 `Grep` 搜索至少 2 个关键词定位受影响代码
- 用 `Read` 读取每个命中文件的完整上下文（≥ 30 行）
- 追溯相关 state 变量、computed 属性、composable 函数链路
- 搜索 `UPDATE_LOG.md` 确认是否有过类似改动
- **每个技术结论必须附带源码证据**
- 记录所有相关文件的绝对路径用于 `file:///` 链接

### 3. 起草规范
创建 `specs/[feature_name]/spec.md`，结构如下：

#### 背景
- 问题/需求来龙去脉
- 技术结论附带 `file:///` 链接

#### 视觉需求
- **必须含 ASCII 布局图**（改动前后对比）
- 禁止仅用文字描述

#### 功能需求
- 根因分析（每条附源码链接）
- 具体修复方案（每个点含目标文件 + 行号 + 改动前后代码对比）
- 现有机制复用清单

#### 涉及文件清单
- Markdown 表格：文件、类型（新增/修改/删除）、说明

#### 边界情况
- ≥ 4 条，每条含：场景 + 风险 + 缓解策略

### 4. 出厂自检
创建后立即用 `Grep` 搜索 `spec.md` 中的 `file:///` 链接数，不足 3 则补充。

### 5. 输出
在回复末尾附带自检表：
```
| 自检项 | 要求 | 实际 | 达标 |
|--------|------|------|------|
| file:/// 链接数 | ≥ 3 | ? | ✅/❌ |
| ASCII 布局图 | 有 | 有/无 | ✅/❌ |
| 代码片段数 | ≥ 2 | ? | ✅/❌ |
| 边界情况条目 | ≥ 4 | ? | ✅/❌ |
| 机制复用说明 | 有 | 有/无 | ✅/❌ |
| 文件清单表格 | 有 | 有/无 | ✅/❌ |
```

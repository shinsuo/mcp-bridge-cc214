# MCP Verify — 功能验证

对已实施的功能进行验证，输出报告。

## 验证标准

| # | 检查项 | 合格标准 |
|---|--------|----------|
| 1 | 环境确认 | 记录 `npm run build` 结果、`dist/` 是否最新 |
| 2 | 视觉审计 | 逐条对比 spec 的视觉需求，给出 ✅/❌ |
| 3 | 边界测试 | 逐条对比 spec 的边界情况，给出结果 |
| 4 | 回归检查 | 确认无已有功能退化 |
| 5 | 最终结论 | 明确给出通过/不通过 |

## 验证指令

### 1. 环境确认
- 执行 `npm run build`，确认编译通过且产物最新
- 如有 MCP bridge 连接，可通过 `ping` 确认服务状态
- 提醒用户在 Cocos Creator 中重新加载插件

### 2. 视觉审计
- 加载 `specs/[feature_name]/spec.md` 的视觉需求章节
- 如有 MCP 连接，使用 `capture_runtime_screenshot` 获取截图
- 逐条对比输出：
```
| # | 视觉需求描述 | 判定 | 差异说明 |
|---|-------------|------|---------|
| 1 | xxx | ✅ 符合 | — |
| 2 | yyy | ❌ 不符 | 具体差异 |
```

### 3. 逻辑审计
- 加载 spec 的边界情况章节
- 逐条转化为测试用例：
```
| # | 边界场景 | 测试步骤 | 期望结果 | 实际结果 | 判定 |
|---|---------|---------|---------|---------|------|
| 1 | xxx | 操作步骤 | 期望 | ? | ✅/❌ |
```
- 检查编辑器控制台是否有新增报错（可使用 `get_runtime_logs`）

### 4. 回归检查
- 确认原有功能未被破坏
- 可使用 `get_runtime_stats` 检查性能是否退化

### 5. 输出报告
创建 `specs/[feature_name]/verify_report.md`：
```markdown
# 验证报告 - [功能名称]

## 测试环境
- 日期：YYYY-MM-DD
- 编译状态：通过/失败
- 编辑器版本：Cocos Creator 2.4.x

## 视觉审计结果
（表格）

## 边界测试结果
（表格）

## 回归检查结果
- [ ] 已有功能正常
- [ ] 控制台无新增报错
- [ ] 性能无退化

## 最终结论
**✅ 通过** / **❌ 不通过**（附原因）
```

### 6. 汇报
- 通过数/总数
- 不通过项的关键问题
- 建议下一步（修复 → 重新验证 / 发布）
